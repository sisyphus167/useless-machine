## 3. API key handling convention (new)

Applies to piece 11 and any future piece that touches a paid external API.

### Rules

- Pieces read keys from environment variables only. No `.env.example`, no repo-level config files, no shared `secrets/` module.
- When a required key is missing, the piece falls back gracefully to a local, zero-dependency imitation. The piece must not raise, crash, or silently hang.
- The piece's README explicitly names the env var, gives a placeholder example (`sk-ant-XXX` style), and warns about cost where applicable.
- Keys are never logged, never printed to stdout or stderr, never embedded in traceback messages, never committed. README examples always use placeholders.

### Rationale

The repo is concept art, not a SaaS. A reader browsing the repo must be able to run any piece without signing up for accounts. But: when the real key is present, the real API is called — the piece's "authentic" state is the real state; the fallback is an acknowledged diminishment.

## 4. Per-piece designs

### 08 wedding-certificate

**Faction**: Shannon (self-negating).

**Thesis**: Two classes that cannot exist apart. Their only operation is marriage; after, nothing.

**Form**: `pieces/08-wedding-certificate/main.py` + `README.md`.

**Key design**:

- Outer class `Wedding`, with inner classes `Bride` and `Groom`.
- Each inner class's `__init__` requires the other as a constructor argument — an unsatisfiable chicken-and-egg for normal construction.
- `Wedding.__new__` resolves the paradox: calls `object.__new__` on each inner class (bypassing their `__init__`), manually sets `.groom` / `.bride` cross-references, returns the tuple `(bride, groom)`.
- `bride, groom = Wedding()` — invoking `Wedding` IS the wedding. Happens exactly once.
- No public methods on `Bride` or `Groom`. No `divorce`, no `communicate`, no `reconsider`.

**Visible behavior**:

```
$ python main.py
a Bride at 0x7f... is married to a Groom at 0x7f...
```

One line. Memory addresses prove they exist; mutual reference proves the marriage; no further output proves nothing else happens.

### 09 ping-pong

**Faction**: Sisyphus (eternal cycle, bounded).

**Thesis**: Two HTTP servers exist only to pass a request back and forth until it dies of exhaustion.

**Form**: `pieces/09-ping-pong/main.py` + `README.md`.

**Key design**:

- Two tiny Flask apps — Server A on `localhost:5001`, Server B on `localhost:5002`.
- Each app defines one catch-all route (any path, any method).
- A's route forwards to B; B's route forwards to A, using `requests` (which is a Flask transitive dep).
- Hop-counting via an `X-Hops` header. Incoming request with no `X-Hops` is treated as 0. Each server increments by 1 before forwarding. If the incoming value ≥ 30, the receiving server returns `508 Loop Detected` with body `hopped {N} times`.
- `main.py` orchestration:
  1. Launches Server A and Server B via `multiprocessing.Process` (separate OS processes, clean shutdown via `terminate()`).
  2. Polls until both ports are bound (short retry loop on socket connect).
  3. Sends a single `GET http://localhost:5001/` with no body.
  4. Prints the final response (status + body).
  5. Terminates both server processes and waits.

**Visible behavior**:

```
$ python main.py
[A] listening on :5001
[B] listening on :5002
→ sending GET to A
← 508 Loop Detected: hopped 30 times
[A] stopped
[B] stopped
```

### 10 solipsist

**Faction**: Over-engineering.

**Thesis**: A "distributed system" with real-sounding protocols (leader election, gossip, cluster health), running as three subprocesses on localhost coordinating via a shared file. The distribution is architectural theatre.

**Form**: `pieces/10-solipsist/main.py` + `pieces/10-solipsist/node.py` + `README.md`.

**Key design**:

- `main.py`:
  - Generates a unique cluster-state file path (`/tmp/solipsist-cluster-<uuid>.jsonl`).
  - Spawns three subprocesses: `python node.py --id 0 --cluster-file <path>`, `--id 1`, `--id 2`.
  - Waits for all three to exit.
  - Prints `CLUSTER STATUS: HEALTHY` if the cluster-state file contains the expected "cluster healthy" marker; otherwise `CLUSTER STATUS: UNKNOWN`.
  - Cleans up the cluster-state file.

- `node.py`:
  - Takes `--id` and `--cluster-file` args.
  - On startup: appends `{"event": "online", "node": N, "ts": ...}` to the cluster file, prints `[node-N] online`.
  - Leader election: reads all current entries in the cluster file; if no `leader` event exists, and all three nodes are online, the node with the highest id writes `{"event": "leader", "node": N}` to the file — becomes leader.
  - The leader writes `{"event": "cluster_healthy"}` after a brief pause, then exits.
  - Non-leaders wait for the `cluster_healthy` event, then exit.

- README top uses a straight-faced paragraph in real distributed-systems vocabulary ("three-node quorum", "byzantine fault tolerance", "gossip-based consensus") without winking. The seriousness IS the joke.

**Visible behavior**:

```
$ python main.py
[node-0] online
[node-1] online
[node-2] online
[node-2] leader election: I am the leader
[node-2] cluster healthy
[node-0] exiting
[node-1] exiting
[node-2] exiting
CLUSTER STATUS: HEALTHY
```

### 11 echo-gpt

**Faction**: Self-reference + LLM.

**Thesis**: A chatbot that rephrases the user's most recent message in different words. Every reply is the user's own words, back at them.

**Form**: `pieces/11-echo-gpt/main.py` + `pieces/11-echo-gpt/fallback.py` + `README.md`.

**Key design**:

- CLI loop in `main.py`:
  1. Check `ANTHROPIC_API_KEY` at startup. If present, use the Claude path; if absent, print a short notice (`(no ANTHROPIC_API_KEY — running in fallback mode)`) and use the fallback path.
  2. Loop: print `you> `, read a line, produce reply, print `it > <reply>`.
  3. Empty input is ignored. `exit`, `quit`, EOF (Ctrl-D), and KeyboardInterrupt (Ctrl-C) cleanly terminate without a traceback.
  4. No conversation memory — each turn is independent. This is faithful: the chatbot's job is rephrasing the most recent message, nothing else.

- Claude path:
  - Model: `claude-haiku-4-5-20251001` (cheap, fast, sufficient for paraphrasing).
  - System prompt: `Paraphrase the user's most recent message in different words. Do not answer their question, do not add new information, do not ask follow-up questions. Only rephrase what they said.`
  - Messages: `[{"role": "user", "content": <user_input>}]` — no history.
  - `max_tokens`: 200.
  - Errors (network, rate limit, API down): caught, fall through to the fallback for that single turn, with a short `(api error — fallback)` notice.

- Fallback (`fallback.py`):
  - Deterministic local rewriter.
  - Algorithm: split on whitespace; if 3+ tokens, shuffle the middle tokens (preserving first and last); prefix the result with `so what you're saying is, `.
  - Deterministic: `random.Random(hash(text))` as the shuffle RNG, so the same input always produces the same output.

- API key discipline:
  - Read via `os.environ.get("ANTHROPIC_API_KEY")`.
  - Never logged, never printed.
  - Tracebacks from API errors are caught before reaching the user — the piece only prints `(api error — fallback)` without exposing the underlying exception text.

**README** (bilingual):

- Run section with env var placeholder: `export ANTHROPIC_API_KEY=sk-ant-XXX`.
- "Without a key" subsection notes the fallback and its diminished effect.
- Cost note: each exchange costs a small fraction of a US cent.

**Visible behavior (with key)**:

```
$ python main.py
you> 今天天气真好
it > 今天的天气令人愉悦
you> 我刚刚说的就是这个意思
it > 你刚才所表达的正是此意
you> exit
```

**Visible behavior (without key)**:

```
$ python main.py
(no ANTHROPIC_API_KEY — running in fallback mode)
you> the sky is blue today
it > so what you're saying is, the sky blue is today
```

### 12 autocomplete-the-autocomplete

**Faction**: Self-reference + Over-engineering.

**Thesis**: A complete Language Server Protocol implementation whose only completion, for any language and any context, is `# autocomplete this`.

**Form**: `pieces/12-autocomplete-the-autocomplete/main.py` + `README.md`.

**Key design**:

- Uses `pygls` to implement a minimal LSP server.
- Registers a completion provider.
  - Trigger characters: `['.', '(', ' ', '\n', ':']` (plus manual invocation).
  - Returns a single `CompletionItem`:
    - `label`: `# autocomplete this`
    - `kind`: `CompletionItemKind.Snippet`
    - `detail`: `a suggestion`
    - `insert_text`: `# autocomplete this`
- Server runs on stdio (the standard LSP mode). `python main.py` launches it; the process speaks LSP over stdin/stdout.
- No language filter at the server level — the editor is responsible for which filetypes to bind it to.
- No other LSP capabilities (no hover, no diagnostics, no go-to-definition). The server advertises only the completion capability.

**README** (bilingual):

- Configuration snippets for three editors:
  - **VS Code**: via the `Generic LSP Client` extension, or a minimal `tasks.json` + `settings.json` example.
  - **Neovim**: `nvim-lspconfig` example with a custom server entry.
  - **Helix**: `languages.toml` example.
- Warning: enabling this as a language server will suppress normal completions for the filetype it's bound to. Recommend using a throwaway filetype for testing.

**Visible behavior**:

Not a console piece. `python main.py` produces no visible output on its own — it's waiting for LSP messages on stdin. The effect is only visible inside an editor: typing any trigger character gives you the same single suggestion forever.

### 13 oracle

**Faction**: Philosophical.

**Thesis**: A machine that agrees with everything. The reader's job is to stop asking.

**Form**: `pieces/13-oracle/main.py` + `README.md`.

**Key design**:

- Main loop: prompt `? `, read any input, print `yes`, repeat.
- Empty input also gets `yes`.
- EOF (Ctrl-D) and KeyboardInterrupt (Ctrl-C) cleanly exit without printing anything.
- Target: 10 lines of Python, maximum.

**Visible behavior**:

```
$ python main.py
? should I quit my job
yes
? is this useless
yes
? are you broken
yes
? ^C
$
```

## 9. Acceptance criteria

Each piece is complete when:

- `README.md` exists in the piece directory, is bilingual (unless a deliberate single-language choice is documented), and conforms to the per-piece anatomy from the first spec.
- The piece's entrypoint (`main.py` or LSP server) runs cleanly on a fresh `pip install -e ".[dev]"` in a Python 3.11+ venv.
- Piece 11 works in both code paths:
  - With `ANTHROPIC_API_KEY` set: at least one successful Claude API round-trip.
  - Without `ANTHROPIC_API_KEY` set: a deterministic fallback response, no crash.
- Piece 12 can be launched and responds correctly to a simple LSP `initialize` handshake (verifiable with a tiny Python LSP client test, run manually — no automated test committed).
- Piece 09 launches both servers, receives the 508 response, and shuts both down without leaking a child process.
- Piece 10 produces the expected cluster-state log and exits cleanly; the cluster-state file is cleaned up.
- Root `README.md` index table is updated with rows 08–13.
- The new dev dependencies are in `pyproject.toml`.
